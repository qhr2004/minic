# 毕业论文目标与项目实现对应关系

本文档用于毕业论文撰写时，将开题阶段的目标与当前 MiniC 项目中的实际实现情况逐项对应。由于仓库中未附开题报告原文，下文按照当前项目已经明确形成的目标链路进行归纳，并严格区分“已实现”“部分实现”“未实现”，避免在论文中夸大成果。

## 1. 总体对应关系

当前项目的主链路可以概括为：

```text
MiniC source
  -> Lexer
  -> Parser
  -> AST
  -> Semantic Analyzer
  -> TAC IR
  -> AST-to-Maple IR lowering
  -> external Ark backend handoff
  -> CTest regression tests
```

从毕业论文角度看，当前项目已经完成“安全子集语言设计 + 编译前端 + 中间表示展示 + Maple IR lowering + 外部后端交接接口 + 基础回归测试”的主体框架；尚未完成的部分主要集中在真实 OpenArkCompiler 后端固定接入、真实可运行的 ABC 验证、运行时与标准库支持。

## 2. 已实现部分

下表可直接对应论文中“系统设计与实现”章节的主要完成项。

| 开题目标方向 | 当前状态 | 项目中的对应实现 | 论文写法建议 |
| --- | --- | --- | --- |
| MiniC 安全子集设计 | 已实现 | 当前语言子集聚焦局部变量、表达式、控制流、函数、`int/float/char` 基本类型，不支持数组、指针、结构体、标准库等高风险或高复杂度特性 | 可写为“面向教学演示与安全子集验证的 MiniC 语言范围设计”，强调有意收缩语言边界，而不是功能缺失 |
| Lexer | 已实现 | 已实现关键字、标识符、整数/浮点/字符字面量、运算符、分隔符、注释处理与行列号定位 | 可写为“完成词法规则定义与手工词法分析器实现，支持错误定位” |
| Parser | 已实现 | 采用递归下降方法实现语法分析，支持函数、块、`if-else`、`while`、`for`、赋值、函数调用、前后缀 `++/--` 等结构 | 可写为“完成递归下降语法分析器，实现从 token 流到 AST 的构建” |
| AST | 已实现 | 已建立统一 AST 节点层次，覆盖语句、表达式、函数与程序节点 | 可写为“完成抽象语法树设计，为语义分析与后续 lowering 提供统一中间结构” |
| Semantic Analyzer | 已实现 | 已实现符号表、作用域栈、重复定义检查、未声明标识符检查、返回类型检查、条件类型检查、实参与形参匹配检查 | 可写为“完成以符号表为核心的语义检查模块，实现 MiniC 子集的静态一致性约束” |
| TAC IR | 已实现 | 已实现字符串形式的三地址码 IR，用于展示前端 lowering 结果与伪代码生成 | 可写为“完成教学展示型 TAC IR 设计与生成”，不要写成“强类型工业级 IR” |
| AST-to-Maple IR lowering | 已实现 | 已实现从 AST 直接生成 Maple IR，支持变量、算术、比较、控制流、函数调用、前后缀 `++/--` 等 | 可写为“完成从抽象语法树到 Maple IR 的直接 lowering” |
| Ark Backend Handoff | 已实现 | `--emit-abc` 已实现为 `MiniC -> .mpl -> external command template -> output path` 的外部交接接口 | 可写为“完成外部 Ark 后端交接机制”，不要写成“已完成真实 Ark 后端集成” |
| CTest 回归测试 | 已实现 | 已建立 `ctest` 体系，覆盖 Maple IR lowering、类型路径、语义错误与 ABC handoff 错误处理等回归项 | 可写为“完成基础自动化回归测试体系，支持关键 lowering 行为与典型错误路径验证” |

## 3. 部分实现部分

这部分建议在论文中单列为“已具备基础能力，但仍存在边界或未完全闭环”的内容。

| 目标方向 | 当前状态 | 当前完成度 | 论文写法建议 |
| --- | --- | --- | --- |
| `float` / `char` 类型路径 | 部分实现 | 前端语法、语义检查与 AST-to-Maple 路径已支持；`float/char` 的声明、赋值、返回和类型检查已有回归测试；但默认 TAC IR 仍是字符串形式，类型保真能力主要依赖 AST-to-Maple 路径 | 可写为“已在前端与 AST-to-Maple 路径中完成基本类型支持，但在弱类型 TAC 路径上仍有限制” |
| Maple IR 类型映射 | 部分实现 | AST-to-Maple 已明确支持 `int -> i32`、`float -> f32`、`char -> i8`；相关 Maple 输出已有回归测试；但 TAC-to-Maple 由于缺少强类型元数据，仍以 `i32` 为主 | 可写为“在 AST 直接 lowering 场景下完成类型映射，在 TAC 复用路径上仍存在退化” |
| ABC 生成链路 | 部分实现 | 已完成 `.mpl` 生成、命令模板替换、外部命令调用、错误处理和中间文件保留；但后端仍为外部 handoff，未固定真实 OpenArkCompiler 工具链 | 可写为“完成 ABC 输出接口与后端交接链路，但尚未完成真实官方后端闭环验证” |

## 4. 未实现部分

这部分应在论文中明确作为“后续工作”或“当前工作边界”，不能写成已经完成。

| 目标方向 | 当前状态 | 当前情况 | 论文中应如何表述 |
| --- | --- | --- | --- |
| 真实 OpenArkCompiler 后端固定接入 | 未实现 | 项目当前依赖外部 `MINIC_ARK_BACKEND_CMD` 模板，仓库内没有固定的真实 OpenArkCompiler 后端实现 | 应写为“预留外部后端接口，尚未完成真实工具链固定接入” |
| 真实可运行 ABC 验证 | 未实现 | 当前无法证明生成结果是已确认可运行的真实 Ark 字节码；若未配置真实后端，输出只能视为 handoff artifact | 应写为“尚未完成真实 ABC 产物的官方工具验证与运行验证” |
| 运行时集成 | 未实现 | 当前没有运行时环境、启动代码或宿主执行链路 | 应写为“尚未集成目标运行时环境” |
| 标准库支持 | 未实现 | 当前不支持标准输入输出、库函数声明与链接语义 | 应写为“尚未实现标准库与外部符号支持” |
| 数组 / 指针 / 结构体 | 未实现 | 当前语言子集未覆盖复合数据类型与相关语义规则 | 应写为“当前工作范围限定为基本类型与结构化控制流，不扩展到复杂数据模型” |

## 5. 论文中可强调的创新点与工作亮点

以下内容可以作为论文中的“设计亮点”或“实现特点”进行强调，但表述时应保持克制，避免写成超出项目范围的通用编译器创新。

### 5.1 面向安全子集的 MiniC 语言设计

- 通过主动收缩语言范围，将实现重点放在可验证、可演示的编译链路上。
- 语言子集覆盖变量、表达式、控制流和函数等核心结构，避免在毕业设计阶段被复杂数据模型和运行时问题牵制。

### 5.2 递归下降前端实现

- 采用手工递归下降方式完成语法分析，结构清晰，适合展示表达式优先级、控制流语法和函数构造过程。
- 该实现便于与 AST 节点设计、语义分析和后续 lowering 逻辑直接对接。

### 5.3 语义检查与符号表

- 使用作用域栈和符号表实现重复定义、未声明标识符、返回类型、条件类型和函数参数匹配等静态检查。
- 对毕业论文而言，这一部分可以体现“从语法正确到语义正确”的编译前端完整性。

### 5.4 AST 到 Maple IR 的 lowering

- 项目不是停留在 AST 或 TAC 展示，而是进一步实现了 AST 到 Maple IR 的 lowering。
- 这一点可以作为论文中“面向目标中间表示的编译后端前半段实现”来强调。

### 5.5 控制流显式基本块生成

- `if-else`、`while`、`for` 在 Maple IR 中被展开为显式基本块与 `brfalse` / `goto` 跳转。
- 这体现了从结构化语法到显式控制流图形态的转换过程，适合在论文中作为关键技术实现展开。

### 5.6 `int/float/char` 到 Maple primitive type 的映射

- AST-to-Maple 路径中，已明确完成 `int -> i32`、`float -> f32`、`char -> i8` 的类型映射。
- 这一点可以作为“类型系统在目标 IR 层面的保真设计”进行论述，但必须同时说明 TAC 路径仍较弱。

### 5.7 外部 Ark 后端 handoff 机制

- 当前项目虽然没有内置真实 OpenArkCompiler 后端，但已经把 `MiniC -> Maple IR -> external backend command` 的交接模式固定下来。
- 这一设计可以被写成“为后续真实工具链接入预留标准接口”，而不应写成“已经打通真实 Ark 后端”。

## 6. 论文撰写时的推荐表述边界

为了避免答辩或评审时出现“实现范围与论文字面表述不一致”的问题，建议使用以下边界：

### 6.1 可以使用的表述

- “实现了 MiniC 安全子集编译器原型”
- “完成了词法、语法、语义分析与 AST 构建”
- “实现了 TAC IR 与 AST-to-Maple IR lowering”
- “实现了面向外部 Ark 后端的 handoff 接口”
- “建立了基于 CTest 的自动化回归测试体系”

### 6.2 不建议使用的表述

- “实现了完整 C 编译器”
- “实现了真实 OpenArkCompiler 后端”
- “已经生成并验证可运行的真实 ABC 字节码”
- “支持标准库和完整运行时”
- “支持复杂 C 数据模型如数组、指针和结构体”

## 7. 建议的论文章节映射

可将当前项目内容大致映射为如下章节：

| 论文章节 | 对应项目内容 |
| --- | --- |

## 8. 最终建议

如果论文要突出项目完成度，建议主叙事放在以下事实上：

- 已完成一个可运行的 MiniC 安全子集编译器原型。
- 已完成从源程序到 AST、语义分析、TAC IR 和 Maple IR 的主要链路。
- 已实现面向外部 Ark 后端的交接机制与基础自动化测试。

如果论文要强调后续工作，建议明确写出：

- 当前项目的“后端”更多体现为 Maple IR lowering 与外部 handoff，而不是已闭环的真实 OpenArkCompiler 后端。
- 真实 ABC 产物验证、运行时集成、标准库支持和复杂数据类型扩展仍属于后续工作。
